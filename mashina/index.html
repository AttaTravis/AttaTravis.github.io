ProjectSchema.InProcessItem

  • Item - NVARCHAR
  • Volume - NVARCHAR
  • Description - NVARCHAR
  • Processed - BIT

In-Process Page

  • Shows list of file types & whether they are in-process or completed
    • Gets each file type and checks for count in-process bit = 0
  • Click them to see completion page

Completion Page

  • Show how many rows currently exist
  • Show how many had exceptions
  • Show current mappings
  • Provide UI to rectify exceptions
    • Delete item
    • Change value

Alter Item

  • Edit all values

Data Validator Service

  • Pull batches of data from Project.InProcessItem
  • Validate Data
  • Stores valid items in ProjectSchema.Item
  • Stores invalid items in ProjectSchema.InvalidItem with relevant error/issue
  • Provides updates to UI via SignalR or places event on rabbitmq and another service serves that to UI via SignalR

Let's Talk SCALE

  • Data validator service could be broken up into multiple services of the same kind with only configuration changes
    • Maybe 1 per file type?
    • I think I like 1 per file type (SKUs, orders, etc.)
  • It could also just reside as a hosted service inside the API
  • These are mostly a question of usage
    • How much is data validator really going to be used that multiples of it would be required.
  • It also needs to understand ProjectSchema

API - Controller - InProcessFiles
